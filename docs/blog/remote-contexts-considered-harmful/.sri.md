> Future versions of this specification may incorporate subresource integrity [[SRI](https://www.w3.org/TR/json-ld11/#bib-sri)] as a means of ensuring that cached and retrieved content matches data retrieved from remote servers; see [:material-github: w3c/json-ld-syntax#86](https://github.com/w3c/json-ld-syntax/issues/86).
>
> — [JSON-LD 1.1 § Security Considerations](https://www.w3.org/TR/json-ld11/#security)

One concrete proposal is [:material-github: w3c/json-ld-syntax#108](https://github.com/w3c/json-ld-syntax/issues/108) (see also [:material-github: w3c/json-ld-syntax#422](https://github.com/w3c/json-ld-syntax/issues/422)):

<div class="grid" markdown>
<div markdown>
```json title="john-lennon-sri.jsonld"
--8<-- "docs/blog/remote-contexts-considered-harmful/john-lennon-sri.jsonld"
```
</div>
<div markdown>
```text title="Result"
--8<-- "docs/blog/remote-contexts-considered-harmful/john-lennon-sri-result.txt:1:6"
```
</div>
</div>

<div class="grid" markdown>
<div markdown>
#### :white_check_mark: Pro
- **Familiar model**: Direct analogy to a proven browser standard, HTML SRI
- **Self-contained reference**: The hash travels with the reference, with no separate lockfile
</div>

<div markdown>
#### :x: Contra

- **Awkward object form**: A plain URL reference turns into a structured object, making the simplest `@context` case heavier and less readable
- **Syntax ambiguity**: It is unclear how to distinguish metadata objects from inline context-by-value objects
- **Incomplete coverage**: Does not yet cover `@import` chains
</div>
</div>
